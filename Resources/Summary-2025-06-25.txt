# Project Summary: Uvf.Net - Universal Vault Format C# Library

## Project Overview
**Uvf.Net** is a comprehensive C# implementation of the Universal Vault Format (UVF) specification, originally ported from a Java library. The project provides client-side encryption and decryption capabilities for secure file storage, particularly designed for cloud storage scenarios where files and filenames need to be encrypted before upload and decrypted upon retrieval.

## Core Components

### 1. UvfLib.Core
- **Purpose**: Core cryptographic implementations and API definitions
- **Key Features**:
  - UVF v3 specification compliance
  - Multi-user JWE vault management with PBKDF2 and Scrypt key derivation
  - File content, filename, and header encryption/decryption
  - Directory metadata handling
  - JSON serialization with AOT-compatible source generators (`UvfJsonContext`)

### 2. UvfLib.Master
- **Purpose**: High-level facade and unified API for end-users
- **Architecture**: Self-contained project that exposes everything needed without requiring additional references
- **Key Features**:
  - `VaultManager` class - primary entry point for vault operations
  - Facade pattern for `KeyDerivationParameters` (hides internal Core API)
  - Support for both UVF and Cryptomator vault formats
  - Streaming operations with configurable buffer sizes
  - Two build modes:
    - **Debug**: Regular managed .NET library (`PublishAot=false`)
    - **Release**: AOT compiled with native C-style exports (`PublishAot=true`)

### 3. UvfLib.Vault
- **Purpose**: Vault-specific operations and file system abstractions
- **Features**: File I/O operations, vault detection, configuration management

### 4. Supporting Projects
- **UvfLib.Tests**: Comprehensive test suite
- **Benchmarks**: Performance testing and optimization
- **Bindings**: Multi-language wrappers (C++, C#, Go, Java, Node.js, PHP, Python)

## Application Projects

### DemoApp (AOT Native Exports)
- **Purpose**: Demonstrates AOT-compiled native exports with C-style API
- **Architecture**: Uses TitanVault wrapper system for P/Invoke interoperability
- **Build Configuration**: 
  - Release mode with `PublishAot=true`
  - Generates `TitanVault.dll` with native C-style exports
  - Requires complex wrapper layer for marshaling

### ExampleVaultApp (Managed Library Testing)
- **Purpose**: Tests core functionality using managed .NET libraries directly
- **Architecture**: References only `UvfLib.Master` for clean API access
- **Key Features**:
  - Direct VaultManager API usage without P/Invoke overhead
  - Password handling with `char[]` arrays for security
  - Support for both PBKDF2 and Scrypt key derivation
  - Streaming operations with 64KB buffers
  - Clean separation from native export complexity

## Recent Development: AOT vs Managed Library Isolation

### Problem Identification
The project needed to isolate whether issues were in:
1. **Core UVF logic** (VaultManager, cryptographic operations)
2. **AOT export layer** (C-style wrappers, P/Invoke marshaling)

### Solution Architecture
Created two parallel testing approaches:

**DemoApp (AOT Path)**:
```
User Code → TitanVault Wrapper → P/Invoke → AOT Exports → VaultManager → Core
```

**ExampleVaultApp (Managed Path)**:
```
User Code → VaultManager → Core (Direct)
```

### Key Fixes Applied

1. **API Modernization**: Updated from `string` to `char[]` for password parameters
2. **Project Reference Cleanup**: ExampleVaultApp only references UvfLib.Master
3. **Facade Pattern**: Created public `KeyDerivationParameters` in Master to hide Core internals
4. **JSON Serialization**: Fixed AOT compatibility issues in VaultHandler
5. **Scrypt Bug Fix**: Resolved multi-user vault decryption for Scrypt recipients
6. **Debug Logging**: Added comprehensive tracing for AOT export debugging

### Testing Results

**ExampleVaultApp (Managed) ✅**:
- PBKDF2 and Scrypt key derivation working perfectly
- UVF and Cryptomator vault operations successful
- All streaming operations functional
- Clean API without wrapper complexity

**DemoApp (AOT) ❌**:
- Persistent issues with JSON serialization in AOT context
- Complex P/Invoke marshaling problems
- Wrapper layer adds debugging complexity

### Current Status
- **Core Library**: Fully functional with both key derivation methods
- **Managed API**: Clean, self-contained, production-ready
- **AOT Exports**: Identified as source of issues, requires JSON serialization fixes
- **Architecture**: Successfully isolated managed library functionality from AOT export problems

This separation allows developers to use the robust managed library directly while the AOT export layer can be debugged and fixed independently.
